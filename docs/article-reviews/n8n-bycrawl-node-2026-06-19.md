# n8n ByCrawl 節點文章評審
日期：2026-06-19
文章網址：https://www.darrelltw.com/n8n-bycrawl-node/
文章標題：n8n ByCrawl 節點教學：簡單好用的台灣爬蟲服務

---

## Phase 1：三角色獨立分析

### 新手行銷人小美

**背景**：剛接觸 n8n 三個月，無工程師背景，立場是「文章對新手不夠友善」。

#### 認可點

1. **quickNav 導覽很貼心**：開頭的快速導覽讓我能跳到想看的段落，不用從頭滾到底，這個設計很好。
2. **FAQ 問題選得不錯**：「不會寫程式也能上手嗎？」這題問到心坎裡，至少讓我知道有門檻，不會直接踩雷。

#### 具體批評

1. **curl 指令區完全嚇跑新手**
   文章裡有 6 個 curl 指令，還要先 `export BYCRAWL_API_KEY`、還有 `jq`。對行銷人來說 terminal 本身就是陌生的，文章只說「把下面指令複製到 terminal 就能測」，但沒解釋 terminal 在哪、怎麼打開。這段雖然有 callout 說「沒裝 jq 也能跑」，但根本的問題是整個 curl 的概念對非技術人就是個黑盒子。

2. **品牌監控案例只說不做**
   「流程：Schedule Trigger（每日 09:00）→ Config 設定關鍵字 → 4 個 ByCrawl 節點平行搜尋 → Code 合併資料 → AI 情緒分析 → Slack + Google Sheets」——這段流程說得很清楚，但截圖只有後半段，而且沒有 workflow 可以下載或 import。作為行銷人，看完不知道要怎麼動手，只知道「喔這個可以做到」。

3. **「什麼時候不該用 ByCrawl？」→ 翻頁解法超過我的理解範圍**
   「要把 ByCrawl 節點換成 HTTP Request 節點，在 URL 帶分頁參數」——對新手來說，HTTP Request 節點是什麼？跟 ByCrawl 節點有什麼不同？直接換節點的動作沒有說明怎麼設定，這裡我會直接放棄。

4. **文章中間那段「不會寫程式也能上手嗎？」說了等於沒說**
   這段明確告訴我「有門檻」，然後說「找我實作和討論」。但作為讀者，我希望知道的是：有沒有更簡單的路？有沒有什麼前置步驟可以幫我降低門檻？這段感覺像是「免責聲明」，不是真的在幫我解決問題。

### 老手工程師阿明

**背景**：爬蟲經驗豐富，用 n8n 兩年，立場是「技術深度不夠，缺少可重用內容」。

#### 認可點

1. **「踩過的坑」是文章最有價值的部分**：pagination 機制的差異整理（Dcard/Threads 用 offset、Reddit 用 after cursor、PTT/104 用 page、591 用 first_row）這種第一手資料，ByCrawl 的文件根本沒寫，看到這段直接省掉我至少 2-3 小時的測試時間。
2. **curl 驗證範本很實用**：在設 n8n workflow 之前先用 curl 確認 API 能抓到想要的資料，這是正確的工作流程。6 個範本涵蓋翻頁邏輯，直接拿來跑。

#### 具體批評

1. **翻頁 callout 說了一半，最關鍵的設定沒說完**
   callout 說「Mode 選『Update a Parameter in Each Request』」，然後就結束了。對我來說，我需要知道的是：Pagination 的 Parameter Name 是什麼？Expression 怎麼寫才能帶入上一頁的 cursor？不同平台（offset vs after）的切換邏輯要怎麼處理？這是最重要的部分，callout 卻是截斷的，感覺沒寫完。

2. **時間格式踩坑提到「加 Set 或 Code 節點統一」，但沒有範例**
   文章整理了 4 個平台時間格式的差異（PTT 是 `"4/20"`、104 是 `"20260414"` YYYYMMDD、Dcard 是 ISO 8601、591 是人類可讀字串）。這是很好的整理，但後面只說「記得加 Set 或 Code 節點統一時間格式」就結束。我需要的是 Code 節點的範例程式，直接 copy-paste 就能用，而不是讓我自己去推敲怎麼 parse `"4/20"` 這種格式。

3. **品牌監控案例的 AI prompt 只說輸出格式，沒說完整 prompt**
   「prompt 要求輸出：情緒分佈（正面/中性/負面比例）、互動數最高的 3 則貼文、一句話趨勢總結」——這是 prompt 的要求，但實際的 prompt 文字是什麼？要怎麼告訴 AI 分析的是從多個平台合併的社群資料？這段留白讓讀者要自己去猜。

4. **4 個平台資料合併的 Code 節點邏輯完全略過**
   流程說「Code 合併資料」，但實際上 4 個 ByCrawl 節點平行執行後，在 n8n 裡合併不同 branch 的資料有細節要處理（需要 Merge 節點或 Code 節點）。這部分直接跳過，有 n8n 使用經驗的人知道這是一個真實的工程問題。

### 部落格編輯小琪

**背景**：熟悉 SEO 和文章結構，立場是「目標讀者定位模糊，結構需調整」。

#### 認可點

1. **Apify vs ByCrawl 比較表做得很好**：六個維度的對比表清晰精準，一眼就能做出選擇，而且最後有明確的選擇指南（要爬台灣平台→ByCrawl，要複雜客製→Apify）。這段幫助讀者排除疑慮，是很有說服力的內容。
2. **費用計算落地**：品牌監控案例把 Credit 換算成每月 1,140 點、對應 Pro 方案 $29/月，這種具體化讓讀者可以直接評估預算，比說「價格合理」好多了。

#### 具體批評

1. **「不會寫程式也能上手嗎？」的位置和定位讓文章目標讀者模糊**
   開頭 callout 說「這篇文章假設你已經有 n8n 環境」，暗示讀者是有基礎的人。但中段又特別設了這個問題，讓新手覺得「我也是目標讀者」，卻沒有對應的新手支援內容。結果兩邊讀者都覺得這篇文章不太完整。如果這篇的主要受眾是有 n8n 基礎的人，那這段應該改成更明確的「適合誰？」說明。

2. **實戰案例缺少可帶走的成品，大幅降低文章分享價值**
   這是轉換率最高的環節，但案例只有文字架構和一張後半段截圖，沒有 workflow template 下載點。根據內容行銷的邏輯，一篇有可下載 workflow 的教學，分享率和回訪率都會明顯更高。description 也提到「實戰案例」，但讀者點進來期待能動手複製，結果只能「參考一下架構」，是個落差。

3. **文章結尾的總結段落太草草**
   「ByCrawl 最大的價值就是超級方便、一把服務就能拿到 15 個以上平台的資料⋯⋯」這段在 500 字的文章裡重複了開頭說過的話，沒有新的資訊，也沒有行動呼籲（除了「歡迎留言」）。一個強結尾應該是：回應讀者最初的痛點（多平台串接頭痛）→ 說明現在能做到什麼 → 具體的下一步（下載 template、試用連結）。

4. **SEO 角度：h2 標題數量夠，但部分標題不夠關鍵字明確**
   「台灣常見平台實測」和「我實測踩過的坑」這兩個 h2 在 SEO 上偏弱，因為沒有包含「ByCrawl」或「n8n」這類核心關鍵字。對比「ByCrawl 節點功能介紹」就明確得多。

## Phase 2：交叉辯論摘要

### 小美 vs 阿明：技術內容的必要性之爭

**小美**：「那段 curl 指令根本看不懂，terminal 是什麼？直接嚇跑新手行銷人。」

**阿明**：「curl 那段是必要的，讓你在花時間設 n8n 之前先確認 API 真的能用。你測過比沒測過好，省掉你設到一半發現資料不符合需求。」

**小美反駁**：「但問題不是 jq，是 curl 這整個概念對行銷人就是陌生的。為什麼不是提供一個 n8n 測試 template，讓我直接在 n8n 裡點一下 Execute 就能驗證？」

**阿明讓步**：「這個點有道理，n8n 的測試 workflow 可以同時服務兩種讀者。但 curl 那段不應該移除，直接加一行『如果你不熟指令，可以跳過這段』就能解決。」

**辯論結果**：阿明的技術立場站得住腳，curl 區本身有價值；但小美的需求有效，解法是在 curl 區前加一行明確的跳過提示，不是刪除。

---

### 阿明 vs 小琪：功能清單 vs 技術深度的取捨

**阿明**：「操作一覽的 tree 格式很好看，但沒有說各平台回傳的欄位有哪些差異，我要串資料時還是要去查文件。」

**小琪**：「那個 tree 清單已經很完整了，如果每個平台都說回傳欄位，文章會超過 5,000 字，讀者很難消化。問題不在深度，在於讀者看完沒有可以直接動手的成品——沒有 workflow 下載，讀完了然後呢？」

**阿明同意**：「確實，缺少可重用的 Code 節點範例（時間格式統一、merge 邏輯）比缺少欄位說明更痛。翻頁 callout 那段說了方向但沒有完整步驟，比沒說更讓人挫折。」

**辯論結果**：兩個問題都成立，但優先級不同。缺少可動手的成品（workflow template、Code 範例）是共同認定的第一優先問題；技術欄位文件是次要問題，可以用外連 ByCrawl API 文件解決。

---

### 小琪 vs 小美：新手需求是否超出這篇文章的服務範圍？

**小琪**：「這篇文章的目標讀者明確是有 n8n 基礎的人，開頭 callout 就寫了『假設你已經有 n8n 環境』，新手的困擾超出這篇的服務範圍。」

**小美**：「但文章中間設了『不會寫程式也能上手嗎？』這個標題，如果不服務新手，為什麼要特別問這個問題？這個標題明示了新手是讀者之一。」

**小琪退讓**：「這是個矛盾。那段的目的應該是『設定預期』，但讀者會解讀成『這個問題有被考慮到，所以文章應該要幫我』。如果要維持那個段落，就要加一個『如果你是新手，下一步可以⋯⋯』的引導；如果不想加，就把那段改成更中性的標題，例如『新手上手時間估計』。」

**辯論結果**：文章定位確實模糊。「不會寫程式也能上手嗎？」這個標題製造了新手期待，但正文沒有相應支援。解法是二選一：加新手引導，或改標題改掉這個期待。

## Phase 3：整合結論

### 文章優點

1. **第一手實測資料說服力強**：pagination 機制差異（各平台用不同參數）、時間欄位格式差異、X 推文要用 Search 而非 Get User Posts——這些都是作者親自打 curl 測試的結果，不是整理文件，這是文章最大的差異化價值。
2. **Apify vs ByCrawl 比較表精準有效**：六個維度的對比加上最後的選擇指南，幫助讀者快速做決定，降低購買猶豫。
3. **費用計算具體落地**：把品牌監控案例換算成每天 38 Credit、每月約 $29 USD，讀者可以直接評估預算，比說「定價合理」有說服力得多。

### 有效批評

辯論後仍站得住腳的批評：

1. **實戰案例沒有可帶走的成品**（三個角色一致認同）：案例只有架構描述和一張局部截圖，沒有可 import 的 workflow JSON。讀者看完知道「可以做」，但不知道「怎麼馬上跑起來」。這是最大的轉換落差。

2. **翻頁 callout 說到一半就結束**（阿明提出、辯論後確認）：callout 說「Mode 選『Update a Parameter in Each Request』」就結束，關鍵的參數設定步驟（Parameter Name 填什麼、Expression 怎麼寫）完全缺席。有個截圖顯示 HTTP Request Pagination 畫面，但搭配的文字說明沒有對應。

3. **時間格式統一只說概念沒有範例**（阿明提出，辯論後確認）：文章整理了 4 種不同的時間格式，這是很有價值的，但後面只說「加 Set 或 Code 節點統一」，沒有範例程式。讀者遇到這個問題時，還是要自己解決。

4. **定位模糊造成兩邊讀者都覺得不完整**（小美與小琪辯論後收斂）：「不會寫程式也能上手嗎？」這個標題製造了新手期待，但文章沒有相應的新手引導，只有「找我實作」。需要二選一：加引導，或改標題。

### 改動建議（按重要性排）

| 優先級 | 改動項目 | 估計成本 | 預期效益 |
|--------|----------|----------|----------|
| P1 | 加品牌監控 workflow template 下載連結（或 n8n template 頁面） | 中（需建 workflow + 測試） | 大幅提升文章可操作性和分享率 |
| P2 | 補完翻頁 callout 的設定步驟（Parameter Name、Expression 範例） | 低（補幾行文字和截圖） | 解決卡點，真正幫到動手的讀者 |
| P3 | 在時間格式踩坑段落加 Code 節點範例程式（4 個平台轉 ISO 8601） | 低（寫 10-15 行 JS） | 讓「踩過的坑」段落從資訊變成可重用資源 |
| P4 | 「不會寫程式也能上手嗎？」改成有行動引導，或改標題降低期待 | 極低（改幾行文字） | 消除定位模糊的困惑 |
| P5 | curl 區前加一行「行銷人可跳過這段，直接用試用版在 n8n 跑」 | 極低 | 讓非技術讀者不被嚇跑，同時保留技術讀者的價值 |

### 一句話總結

文章的第一手實測資料很有說服力，但缺少可直接帶走的成品（workflow template + 完整的翻頁設定範例），讀者讀完知道「ByCrawl 好用」，卻不知道「怎麼馬上跑起來」。
