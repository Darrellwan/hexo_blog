# n8n 教學文章寫作風格指引

這份文件整理目前多篇 n8n 教學文章（如 Aggregate/Split Out、Structured Output Parser、Webhook、LINE Messaging 社群節點、Filter 等）的共同風格，之後撰寫或改寫 n8n 相關文章時可以當作參考基準。

---

## 1. 目標讀者與定位

- 讀者通常：
  - 已經會基本操作 n8n（知道什麼是 workflow / node）
  - 想解決「某個具體問題」或學會「某個節點／功能」
  - 習慣邊看文章邊照著操作
- 文章目的：
  - 幫讀者「快速成功跑出第一個例子」
  - 順便分享踩雷經驗與設定建議
  - 比官方文件更白話、更貼近實務情境

---

## 2. 語氣與口吻

- 使用「你」「我們」，口語、友善、有點聊天感：
  - 例：`如果你曾經遇到以下困擾：`、`我們先來看最常見的場景`
- 先用白話解釋，再給技術名詞：
  - 例：`Webhook 用白話文來說就是一個接收器，Webhook = Web + Hook，可以想像成網路世界的釣竿或鉤子`
- 可以穿插個人經驗和感想，但保持精簡：
  - 例：`這個節點真的是我好想早一點學會的節點！`
- 避免太學術或太官方的語氣，用「實測」「踩雷」「建議」帶出觀點：
  - `實測下來…`、`我自己的建議是…`、`這裡很容易踩雷，要特別注意`

---

## 3. 文章標題與 Front Matter 習慣

- 標題格式：
  - 節點教學類：`n8n XXX 節點教學 - 一句話總結功能`
    - 例：`n8n Filter 節點教學 - 資料篩選與條件判斷`
  - 實戰流程類：`n8n XXX 實戰 - 用 YYY 解決 ZZZ 問題`
    - 例：`n8n 資料處理踩雷 - Aggregate 合併與 Split Out 分割實測`
- Front matter 重點欄位：
  - `title`：如上格式
  - `tags`：`n8n` + `n8n節點介紹` / `n8n教學` / 功能關鍵字
  - `categories`：通常為 `n8n`
  - `description`：一句話講出這篇文章要解決什麼問題、包含哪些內容

---

## 4. 文章基本結構模板

大部分 n8n 教學文可以沿用這個骨架：

1. 封面圖
   - `{% darrellImageCover ... %}`
2. （選用）快速導覽區塊
   - 預計閱讀時間
   - 適合對象
   - 你將學到
   - QuickNav（可快速跳轉段落）
3. 為什麼需要這個節點／功能
   - 從「痛點／踩雷」出發，而不是直接講介面
4. 安裝或前置準備（如果需要）
   - 版本需求、Beta 開關、Credential、API key 等
5. 基本設定教學
   - 一步一步＋截圖＋簡短說明
6. 功能與設定詳解
   - 每個模式／功能一個小節，搭配圖
7. 1–2 個實戰案例
   - 用完整 workflow 展示實際使用情境
8. 常見問題、限制與建議
9. 結語與相關文章推薦

可以視篇幅、複雜度決定要不要出現「快速導覽」「實戰案例」「FAQ」等區塊。

---

## 5. 快速導覽與 QuickNav

- 較長的文章（例如 Guardrails、LINE Messaging 社群節點）建議加上「快速導覽」區塊：
  - `預計閱讀時間`：給一個範圍（例：5–8 分鐘）
  - `適合對象`：一行描述讀者條件
  - `你將學到`：3–4 個 bullet，列出主要學習成果
- `QuickNav` 用來協助快速跳段，習慣格式：

```liquid
{% quickNav %}
[
  {
    "text": "節點安裝設定",
    "anchor": "n8n-setup",
    "desc": "社群節點安裝、Credential 建立、測試串接"
  },
  {
    "text": "4 大核心功能",
    "anchor": "features",
    "desc": "Send、Multicast、Reply、Get Profile"
  }
]
{% endquickNav %}
```

## 6. 說明方式與段落風格

### 6.1 概念說明

- 先用一句話定義：
  - `Filter 節點會用來篩選資料…`
  - `Aggregate 的用法會是把多個 items 合併為一個 item`
- 搭配生活或直覺比喻：
  - Webhook = 網路釣竿 / 鉤子
- 再補上技術細節與適合場景：
  - `適合用在…`、`很常見的情境是…`

### 6.2 設定教學

- 使用步驟列表：
  - `1. 建立一個 XXX 節點`
  - `2. 在 credential 那邊選取新增一個 credential`
- 每個步驟後面對應一張截圖（`{% darrellImage800 ... %}`）或簡短說明。
- 重要差異點用小節或粗體標出：
  - `### 模式: Individual Fields`
  - `### 進階設定 - 多個 and 或是 or 條件`

### 6.3 實戰案例

- Workflow 先用一句話描述用途：
  - `假設我們今天要在 n8n 做一個新聞摘要的 workflow`
- 然後用「1. 產生 prompt → 2. 加入 AI Agent 節點 → 3. 新增 Structured Output Parser 節點…」這種流水式步驟，讓讀者可以跟著做。
- 圖片重點是「畫面長什麼樣／哪些欄位要填」，文字負責補充「為什麼這樣填」。

---

## 7. 小提醒與踩雷寫法

- 常見用法：
  - `重要提醒：`、`⚠️ 注意：`、`這邊很容易踩雷`
- 內容類型：
  - API 限制（例如：Reply Token 的時效、DataTables 沒有備份）
  - 本機／網域問題（例如 localhost 不能拿來當 webhook 網域）
  - 欄位設定會造成的後果（例如刪除資料不可回復）
- 語氣維持友善且具體：
  - `目前 Data Tables 沒有提供資料備份或是救援的機制，如果誤刪會真的不見，建議刪除前要三思。`

---

## 8. 結語與相關文章

- 結語維持簡短，通常包含：
  - 這個節點／功能「最適合的幾種情境」
  - 個人一句感想或未來補充方向
    - 例：`等未來遇到相關的問題和經驗時，會再來補充`
- 文末加上 2–3 個 `{% articleCard %}` 做導流：
  - 連到相關節點教學（Webhook ↔ LINE Messaging）
  - 連到基礎概念文（If/Switch、內建變數等）

---

## 9. 寫新 n8n 文章時的簡易 Checklist

- [ ] 標題是否符合「節點教學 / 實戰」命名慣例？
- [ ] 有一段「為什麼需要這個節點／功能」從痛點出發？
- [ ] 是否用步驟＋截圖說明「基本設定」？
- [ ] 是否至少有一個「實戰案例」或實測情境？
- [ ] 有標出 1–2 個容易踩雷的地方與建議做法？
- [ ] 文末是否提供相關文章連結，方便延伸閱讀？

這份指引主要針對 n8n 相關內容，但也可以作為其他技術教學文章的參考基礎，在新功能或新節點出現時，保持一致的閱讀體驗與語氣。
